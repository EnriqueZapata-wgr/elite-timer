## 🗜️ Compresión SEGURA (sin afectar otras imágenes)

Tu inquietud es válida: si corres `npm run optimize-images` el script recorre TODO `assets/images/` y aunque skipea <200KB, las que están entre 200-500KB pueden perder 1-15% más cada pasada.

**3 opciones — elige según preferencia:**

### Opción A (más segura, quirúrgica) — comprimir SOLO la nueva carpeta

Edita temporalmente el script `scripts/optimize-images.js` línea 4:

```javascript
// Cambia esta línea:
const TARGET_DIR = 'assets/images';

// A esta (solo durante el run):
const TARGET_DIR = 'assets/images/hoy-extra/tu-dia';
```

Corres:
```powershell
npm run optimize-images
```

**Después RESTAURA** la línea original a `'assets/images'` (importante, no commitees con el cambio temporal).

Las nuevas se comprimirán dramáticamente (5MB → 500KB típico para imágenes MJ B/N). Las viejas NO se tocan.

### Opción B (one-liner sin tocar script) — sharp directo

```powershell
node -e "const sharp=require('sharp');const fs=require('fs');const path=require('path');async function walk(dir){if(!fs.existsSync(dir))return;const items=fs.readdirSync(dir);for(const item of items){const full=path.join(dir,item);const stats=fs.statSync(full);if(stats.isDirectory()){await walk(full);}else if(/\.(png|jpg|jpeg)$/i.test(item)){if(stats.size<200*1024){console.log('skip',full);continue;}const buf=await sharp(full).resize(2048,null,{withoutEnlargement:true}).png({quality:85,palette:true}).toBuffer();fs.writeFileSync(full,buf);console.log('✓',item,Math.round(stats.size/1024)+'KB→'+Math.round(buf.length/1024)+'KB');}}}walk('assets/images/hoy-extra/tu-dia').then(()=>console.log('done'));"
```

(Sí, está feo pero es one-liner. Solo procesa `hoy-extra/tu-dia/**`. Cero riesgo a otras carpetas.)

### Opción C (aceptable, no quirúrgica) — script completo

```powershell
npm run optimize-images
```

Las nuevas se comprimen bien. Las viejas pierden marginalmente (impacto visual mínimo en imágenes B/N porque la palette quantization tiene techo natural — después de 2-3 pasadas, ya no comprime más).

**Recomendación: Opción A.** Es quirúrgica + sin riesgo. El "no commitees el cambio temporal" es importante — si quieres futuro-proof, después podemos agregar argumento `--target` al script en otro mini-fix.
