## 2. Tareas quirúrgicas

### 2.1 AGUA: optimistic override en quickActions (CRÍTICO)

Causa diagnosticada: race condition. `addWater()` emite `day_changed` antes de que Supabase confirme INSERT → day-compiler recompila con valor viejo → re-render parpadea (sube y baja) + tarda 3-5s.

Archivo: `src/components/hoy/HoyEditorialSection.tsx`

**Fix:** mismo patrón optimistic que ya implementaste para booleanos (línea ~191), aplicado a `water.displayCurrent` y `water.current`:

1. Agregar state local `optimisticWaterMl` (number | null)
2. En `quickActions[].onTap`, calcular nuevo total optimista inmediato y `setOptimisticWaterMl(newTotal)`
3. Luego `addWater(userId, delta)` async, NO bloquear
4. Render usa: `const currentMl = optimisticWaterMl ?? water.current`
5. Cuando llegue evento `day_changed` o el next compile, reconciliar:
   - Si `water.current === optimisticWaterMl` → limpiar override (alineados)
   - Si difiere → mantener override (writes en vuelo)
6. Si falla addWater → rollback `setOptimisticWaterMl(prev)` con toast/log

Mostrar también la barra de progreso con valor optimista (no esperar recompile).

### 2.3 Journal NO guarda: debug + UI specific (CRÍTICO)

Causa probable diagnosticada: error genérico oculta el real. Tres causas posibles:
- RLS policy en `journal_entries` rechaza INSERT
- `user.id` es null mid-save
- Network real (poco probable porque otras escrituras funcionan)

Archivo: `app/journal.tsx:243-260`

**Cambios:**

1. **Loguear el error REAL** antes del catch genérico:
```typescript
} catch (insertError: any) {
  console.error('[journal handleSave] insert failed:', insertError?.message, insertError);
  logError('[journal] handleSave failed', insertError);
  // diferenciar
  const msg = insertError?.message ?? '';
  const isAuth = !user?.id || msg.includes('JWT') || msg.includes('auth');
  const isRLS = msg.includes('policy') || msg.includes('RLS');
  const isNet = msg.includes('Network') || msg.includes('Failed to fetch');
  Alert.alert(
    'No se pudo guardar',
    isAuth ? 'Tu sesión expiró. Vuelve a iniciar.' :
    isRLS ? 'Permiso denegado. Reporta a soporte.' :

```

2. **Verificar RLS** en `journal_entries` table — si la causa raíz es RLS, escribir migración idempotente:
```sql
-- Verificar policy existe; si no, crear
DO $$ BEGIN
  IF NOT EXISTS (SELECT 1 FROM pg_policies WHERE tablename = 'journal_entries' AND policyname = 'journal_entries_insert_own') THEN

END $$;
```

3. Aplicar con `npx supabase db push` si la causa es RLS.
